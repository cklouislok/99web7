Abstract

Using ArcGIS Maps SDK for JavaScript 4.29 and adapting their sample codes, created a web application for exploring Toronto Neighborhood Crime Statistics.

Main function 1: Query statistics by geometry

The neighborhood crime statistics are queried by year and crime type among neighborhoods intersecting a buffer, and displayed in a bar chart. The user can drag the center point of the buffer or resize the buffer by dragging the edge point. This function largely fulfils the specification of a proposal deliverable in allowing users to define the location and area of a circle to explore relevant crime statistics.

Main function 2: Option to define custom class breaks for each crime type

By reading the same neighborhood crime statistics, a class breaks renderer is generated using the Feature Layer and the fields of crime types. By clicking the expandable widget, the user can customize the class breaks by selecting the type of crime, the classification method, and the number of breaks. If manual classification method is chosen, the user can specify the cut-off points for each class.

Other functions

Other functions combined offer common functionalities of web maps. They include a number of widgets: the search bar that allows the user to navigate to any location within Toronto, an option to switch the base map, north arrow button to readjust the orientation of the map, a home button to return to the initial map extent, and a full screen button. Particularly, the widgets for the main functions and also legend can be expanded and collapsed for better user experience and clean presentation.

Tips and tricks for configuring a solution involving SDK

Understanding codes:

  • You do not have to understand every single part of the sample code to make things work; if the sample or the desired result is simple, it can be as easy as changing the data source from an example
  • When trying to understand how a code sample works, search where the function or variable appears and works in the code

Working with codes:

  • Sometimes it is better to only make one change at a time to easily identify the issue
  • Always be wary of the indentation and sequence of the codes, especially when you combine code examples/handle huge blocks of codes
  • Considering time constraints, when adapting sample codes, try to stick to the sample or make as few necessary changes as possible to minimize chances of breaking things

Concepts Learnt

  • The major processes in getting SDK to work share similarity with the ETL framework: to extract the desired data from the source, which is the neighborhood crime statistics in our case (using ArcGIS REST Endpoint URL); to transform the data into a format that can be loaded for display, which is summing up the crime statistics by type and year; lastly to load the data to the display, which is the bar chart structure in our case.

Future Possibilities to Improve

Technical challenges

  • Targeting/modifying the right parts of the code to fix things or make things work
  • A fix for a certain issue may lead to another issue
  • Understanding/Dissecting how the sample works

Mistakes Made

  • Not targeting the right code
  • Being too aggressive: making too many changes at once/trying to add significant amount of codes for adding a certain feature; that bounds to fail (unless you are really good with coding)
